カテゴリ 内容表示 ＃ 07
機能名称 言語依存のテキスト表示
説明 縦書き，ルビ，禁則処理など
実現方法
  • 縦書き：CSSを用いる． -webkit-writing-mode: vertical-rl;
  • 右綴じ：content.opfの各ページの順番を設定する<spine> に，page-progression-direction="rtl" (rtlは”right to left”を意味する)を加える．
  • ルビ表示：HTMLタグの<ruby>を用いる．
  • 禁則処理：ePubリーダーにより，自動で行われる．
実現例
  • 縦書き
    • どの文字が禁則処理されるのかをテストしてみます。ブラウザの幅や文字サイズ を調整することで、禁則処理が行われているかチェックできます。全角（カッコ）のテスト、半角(カッコ)のテスト。次に「このカッコ」のテスト。ちぃさぃ文字のテストと、のばーーーすテスト。半角!と全角！のビックリマーク。次に"半角"と”全角”のダブルクォーテーション。云々とか我々とかの小さい「々」のテスト。例えば…とか・・・のテスト。最後に\5,000とか￥５，０００とか100%とか１００％のテスト。
  • ルビ表示
    • 親文字の一文字ごとに対応したルビ（モノルビ）
    平ひら仮が名な 漢かん字じ
    二文字以上の親文字列にまとめてルビ（グループルビ）
    平仮名ひらがな 漢字かんじ
検証結果 環境 結果 原因
iBooks3 (iPad2) ○
Himawari Reader (SONY Tablet S) △ 右綴じのみ不可。
Reader by SONY (SONY Tablet S) △ 縦書きのみ不可。
Kindle (Kindle PaperWhite) × .mobiでは不可。
Readium (Chrome add-on) ○
Calibre △ 右綴じ、縦書き不可。

ePubにおける一時データの保存方法の選択肢と比較
  1. FileSystemAPI ：ファイル操作をする仕組みです。ファイルを読み込んだり、新しくテキストファイルを作成したりすることが出来ます。OS内のファイルを直接やりとりすることは不可能なため、ブラウザ内のLocalStorageの領域を使ってファイル操作を行っているようです。これを利用して”電子教科書名.txt”というテキストファイルを作成し、学習情報を保存していくという考えです。保存するファイル名を”電子教科書名.txt”としているのは、”log.txt”といった共通の名称にすると、複数の電子教科書で保存する情報が競合してしまうためです。競合を防ぐ別の解決策としてはFileSystemAPIのファイル操作で電子教科書ごとにディレクトリを作成し、その下に”log.txt”を置くという方法があります。 　Google Chromeアドオンの電子書籍リーダー(Readium)を用いて、ePubからtxtファイルの保存ができることを実装で確認しました。このtxtファイルを読み出し、新たに情報を追加できることも確認済みです。また、未確認ですが文書ファイル以外（画像など）も保存できると考えられ、有効利用できないか検討中です。 問題点は、FileSystemAPIがGoogle Chromeでしか動作しないことです。そのため、Readiumでしか利用出来ず、汎用性のある仕組みとは言えません。また今後、この仕組みがなくなる可能性も十分に考えられます。
  2.  LocalStorage 　ブラウザ（電子書籍リーダー）内に情報を保存する仕組みです。多くのブラウザで利用可能です。FileSystemAPIでも使っていたこのLocalStorageに直接、情報を保存した方が汎用性のある仕組みになりそうです。FileSystemAPIのようなディレクトリ操作は出来ないものの、電子教科書ごとに別の領域で保存されるため、同じキーを使っても別の電子教科書と競合しません（iBooksで確認）。 　問題点は、保存する情報量が多くなると読み書きにかかる時間も増加し、操作性が悪化する可能性があることです。LocalStorageはディレクトリ操作が出来ず、全ての情報を同じ階層に保存しています。そのため、保存する情報が多いと、情報を取り出すのに多大な時間がかかることが懸念されます。
  3.  WebSQL 　LocalStorageと同様、情報をブラウザ（電子書籍リーダー）内に保存する仕組みです。SQLiteベースのSQLによってデータにアクセスします。これを利用することで、情報をデータベースで電子教科書ごとに管理することができ、保存する情報量が増えても、読み書きの操作がLocalStorageより早くなると予想しています。 　問題点は、WebSQLがW3Cの定めるHTML5の仕様から廃止されていることです。WebSQLではなく、次に示すIndexed Database APIを使用することが推奨されています。 
  4. Indexed Database API 　LocalStorageと同様、情報をブラウザ（電子書籍リーダー）内に保存する仕組みです。WebSQLと異なり、Indexed Database APIはSQLに依存しないNoSQLによってデータにアクセスします。これを利用することで、WebSQLと同様に情報をデータベースで電子教科書ごとに管理できるようになると考えています。問題点は、非常に新しい仕組みでまだ仕様が確定しておらず、各ブラウザがサポートしている確証がないということです。 　以上の4方法で検討を進めており、現状では1. FileSystemAPIを用いて作成した”log.txt”ファイルの読み書きが出来ることをReadium（Google Chromeアドオンの電子書籍リーダー）で確認した段階です。
